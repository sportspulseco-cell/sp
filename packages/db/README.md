# @sportspulse/db

Drizzle-owned data access layer.

## Source of truth

- **Schema:** [`src/schema/`](src/schema) — TypeScript, the canonical model.
- **Migrations:** [`migrations/`](migrations) — generated by `drizzle-kit generate`. Hand-written prelude lives in `0000_init.sql`.
- **Triggers / RLS / functions:** raw SQL in custom migration files, generated via `pnpm generate:custom --name=<x>` and edited.

## Workflow

```bash
pnpm install

# 1. Edit src/schema/*.ts
# 2. Generate a migration from the diff
pnpm generate

# 3. (Optional) write a custom migration for triggers / RLS / functions
pnpm generate:custom --name=rls_iam

# 4. Apply migrations
pnpm migrate

# 5. (One-time) seed reference data
pnpm seed
```

## Connection strings

| Use | Var | Port | Notes |
|---|---|---|---|
| `drizzle-kit`, `migrate.ts`, `seed/index.ts` | `DATABASE_URL` | 5432 | Direct, supports DDL + prepared statements |
| App runtime (NestJS, Next.js) | `DATABASE_POOL_URL` | 6543 | Supavisor pooler, transaction mode, `prepare: false` |

## Why hand-written prelude (`0000_init.sql`)?

Drizzle's TypeScript schema doesn't model:
- `CREATE EXTENSION` (pg_graphql, citext, btree_gist, pgcrypto)
- Trigger functions and triggers (`set_updated_at`, `handle_new_user`, etc.)
- RLS policies (use `pnpm generate:custom` after table migrations)

These ride in raw-SQL migration files alongside the generated ones. Drizzle's migrator runs them in lexical order.

## What is NOT in this package

- Supabase auth schema (`auth.users` etc.) — managed by Supabase. We declare a stub in `src/schema/auth.ts` only so other tables can take FK references.
- Storage / Realtime schemas — same reason.

## Don't apply migrations any other way

`drizzle-kit` and `migrate.ts` are the only migration paths. Do not `apply_migration` via Supabase MCP, Studio SQL Editor, or `psql` against this database — divergence will silently break `drizzle-kit check`.
